Prompt Caching in LLM-Based Agentic Systems

In agentic systems, agents repeatedly send large system prompts, tool schemas, and retrieval results to the model. Without caching, every call re-processes tokens from scratch — paying full latency and compute cost each time.

# How It Works

The KV (key-value) cache in a transformer stores the attention keys and values computed for each token. Prompt caching reuses the KV cache for the static prefix of a prompt (system instructions, tool definitions, RAG chunks) across multiple requests. Only the new, dynamic suffix (the user message or tool result) gets newly computed.

# What Gets Cached vs. Recomputed

Part of Prompt Type Treatment
System prompt Static Cached — KV reused
Tool schemas Static Cached — KV reused
RAG context Static Cached — KV reused
User message Dynamic Recomputed every call
Tool results Dynamic Recomputed every call

# Key Considerations

  • The static prefix must be placed before any dynamic content — the cache breaks the moment ordering changes.
  • Cache TTL is typically 5–10 minutes on hosted APIs. Multi-turn agents that pause between steps may miss the window.
  • Anthropic’s prompt caching (via the cache_control header) can yield up to 90% cost reduction and ~85% latency reduction on cached tokens.

# Best Practices for Agentic Systems

  • Cache: system prompt + tool definitions + any large retrieved context.
  • Leave the conversation history and tool results in the dynamic suffix.
  • Ensure the static prefix is stable across turns — avoid injecting dynamic values (timestamps, session IDs) into the cached region.
  • Monitor cache hit rates; a low rate usually indicates inconsistent prefix ordering.

# Impact

In a typical agentic loop with a large system prompt and tool schemas, prompt caching alone can reduce per-step token costs by 80%+, making multi-step agent workflows economically viable at scale.
