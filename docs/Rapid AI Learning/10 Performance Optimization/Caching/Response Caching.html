Response Caching

While prompt caching saves compute within a call, response caching saves compute across calls by memoizing the model’s full output for a given input. If the exact prompt was seen before, the stored response is returned without touching the model at all.

# How It Works

A hash (usually of the full prompt, or a semantic embedding) is used as a cache key. On a cache hit, the response is returned directly from storage in ~1ms. On a miss, the model runs inference and the result is written to the cache for future use.

# Cache Strategies

# Exact Match Caching

Hash the raw prompt string as the cache key. Zero false positives — a single character difference causes a miss. Best for: templated queries, repeated tool call parameters, FAQ-style pipelines.

# Semantic Caching

Embed the prompt and find nearest neighbours in a vector store. Returns a cached response if cosine similarity exceeds a threshold. More flexible, but risks returning a slightly wrong answer for similar-but-not-identical queries. Requires careful threshold tuning.

# When to Use Each Strategy

Scenario Recommended Strategy
Repeated templated queries Exact match
FAQ / knowledge base lookups Semantic
Tool calls with stable parameters Exact match
User paraphrasing the same question Semantic
Time-sensitive queries (news, prices) Do not cache, or very short TTL

# Cache Invalidation Considerations

  • TTL-based: Responses expire after a fixed duration. Simple but may serve stale data.
  • Event-based: Cache is invalidated when underlying data changes (e.g., a document is updated).
  • Bypass entirely: Time-sensitive tool results (web search, stock prices, live APIs) should never be cached, or use a TTL of seconds.

# Best Practices for Agentic Systems

  • Cache tool call responses when the parameters are stable and the result is deterministic (e.g., get_country_capital("France")).
  • Avoid caching responses from tools that hit live data sources unless you’re comfortable with potential staleness.
  • When multiple parallel agent branches ask the same sub-question, semantic caching prevents redundant inference.
  • Log cache hit/miss rates per tool type — this reveals which tool calls are most worth caching.

# Impact

Response caching can eliminate model calls entirely for known inputs. In pipelines with high query repetition — e.g., multi-agent systems processing similar documents — cache hit rates of 30–60% are achievable, translating directly to cost and latency savings.
