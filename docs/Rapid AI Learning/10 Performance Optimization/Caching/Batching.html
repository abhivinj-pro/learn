Batching

A single LLM forward pass is expensive. GPU utilization during single-request serving is often below 30%. Batching groups multiple requests into one forward pass, amortizing the cost of matrix multiplications across many requests simultaneously.

# The Three Batching Approaches

# Static Batching

Simple and easy to implement. All requests in the batch must finish the longest sequence before any results are returned. Padding waste can be severe (40–60% of compute) when request lengths vary, which they always do in agentic systems. Suitable only for workloads with uniform sequence lengths.

# Continuous Batching (Iteration-Level Scheduling)

Pioneered by the Orca paper and now standard in vLLM, TGI, and TensorRT-LLM. The key insight is that each decoding step (producing one token) is independent. A completed request frees a slot immediately, and a new request can be inserted at the next step. This keeps GPU occupancy near 100% and dramatically reduces P99 latency for queued requests.

# vLLM’s PagedAttention

Goes even further by managing the KV cache like OS virtual memory pages. Sequences can be non-contiguous in GPU memory, eliminating KV cache fragmentation. Enables 2–4× more concurrent sequences than naive continuous batching on the same hardware.

# Async / Offline Batching

Used in data processing pipelines, evaluation harnesses, and non-real-time agentic workflows. Requests are accumulated in a queue, flushed as a batch every N milliseconds or N items, and results handled asynchronously via callbacks or a results queue.

Anthropic’s Message Batches API works this way — offering a 50% cost reduction for non-real-time workloads.

# Comparison

Strategy GPU Utilisation Padding Waste Latency Best For
Static batching Medium High High Uniform-length workloads
Continuous batching Very high None Low Real-time agentic serving
PagedAttention Near 100% None Very low High-concurrency serving
Async / offline Medium Low Acceptable Pipelines, eval, batch jobs

# Best Practices for Agentic Systems

  • Use continuous batching for any real-time agentic serving — it is the default in all major inference frameworks (vLLM, TGI).
  • Use async batching for agentic pipelines where results do not need to stream back to a user in real time (e.g., document processing, evaluation, scheduled tasks).
  • Avoid static batching in agentic contexts — the variable length of agent steps (short tool calls vs. long reasoning traces) makes padding waste especially severe.
  • Profile batch sizes carefully: a batch that is too large increases memory pressure and causes OOM errors; too small leaves the GPU underutilised.

# Impact

Continuous batching combined with PagedAttention can improve throughput by 10–20× compared to naïve single-request serving, and is often the single highest-leverage infrastructure change in a production agentic deployment.
