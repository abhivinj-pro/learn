# A2A (Agent-to-Agent) Protocol — Interview Cheat Sheet

# Core Concept

A2A is a protocol where a master orchestrator agent delegates tasks to specialized sub-agents over HTTP, rather than calling granular functions. Each sub-agent is a black box — the orchestrator only knows what an agent does, not how.

# Architecture in One Line

Orchestrator → discovers agents via Agent Cards → wraps them as tools → delegates natural language tasks → receives normalized string results.

# The 4 Phases (End-to-End)

# Phase 1 — Agent Card Discovery

Each sub-agent self-describes at a well-known URL:

GET http://localhost:8001/.well-known/agent.json

The card contains: name, description, url, capabilities, and skills[].

Key fields to remember:

{
  "capabilities": { "streaming": true, "pushNotifications": false },
  "skills": [{ "id": "search_flights", "inputModes": ["text"], "outputModes": ["text", "data"] }]
}

# Phase 2 — Orchestrator Setup (bind_tools)

Each entire agent is wrapped as a single LangChain @tool — not individual skills.

@tool
async def delegate_to_travel(task: str) -> str:
    """Delegate travel planning tasks — flights, hotels, itineraries."""
    return await send_a2a_task("http://localhost:8001", task)

orchestrator = llm.bind_tools([delegate_to_travel, delegate_to_booker])

Critical: The LLM gets one schema per agent, not N granular schemas like MCP.

# Phase 3 — Wire Protocol (tasks/send)

Every delegation is a JSON-RPC 2.0 POST:

payload = {
    "jsonrpc": "2.0",
    "method": "tasks/send",       # ← A2A standard method
    "id": "req-001",
    "params": {
        "id": "task-abc-123",     # unique task ID
        "message": {
            "role": "user",
            "parts": [{"type": "text", "text": task}]   # natural language
        }
    }
}

Response structure (artifact):

{ "result": { "artifacts": [{ "parts": [{ "type": "text", "text": "BA112, $340" }] }] } }

# Phase 4 — Response Normalization

A2A returns nested artifacts. Must flatten to plain string before the LLM sees it:

def extract_a2a_result(response: dict) -> str:
    artifacts = response["result"]["artifacts"]
    for part in artifacts[0]["parts"]:
        if part["type"] == "text":
            return part["text"]
        elif part["type"] == "data":
            return str(part["data"])
    return response.get("error", {}).get("message", "Unknown A2A error")

# A2A vs MCP — The Critical Distinction

MCP A2A
Tool granularity N fine-grained schemas 1 schema per agent
Invocation Typed args to specific function Natural language string
Sub-agent internals Orchestrator sees the call chain Completely hidden (black box)
Coupling Tight Loose
bind_tools receives search(), insert(), query() … delegate_to_agent(task: str)

They complement each other: A sub-agent in A2A can internally use MCP tools — the orchestrator never knows.

# Keywords to Remember

Term What It Means
agent.json Agent Card — self-description of agent’s capabilities
/.well-known/ Standard path for serving the card
tasks/send JSON-RPC method to send a task to a sub-agent
artifacts A2A response container (equivalent of return value)
parts Content inside an artifact — type: text or type: data
skills[] Named capabilities listed in the Agent Card
inputModes/outputModes Supported modalities per skill (e.g., text, data)
extract_a2a_result() Normalization wrapper — converts artifact → plain string
bind_tools Registers agent-wrappers with the orchestrator LLM
jsonrpc: "2.0" Wire protocol version

# Connection Theory (Important)

  • Transport: Plain HTTP/HTTPS POST to the agent’s root URL (/)
  • Protocol: JSON-RPC 2.0 — every request has jsonrpc, method, id, params
  • Discovery: Agents are found by fetching /.well-known/agent.json — no central registry
  • Async-capable: Agent Cards advertise streaming: true (SSE) or pushNotifications
  • Stateless delegation: Each tasks/send is self-contained; task ID is client-generated UUID
  • Error handling: JSON-RPC errors surface in response["error"]["message"] — normalize these too

# Interview Q&A — Senior AI Engineer Level

Q1: What is the fundamental difference between how MCP and A2A expose agent capabilities to an orchestrator?

MCP exposes granular, typed function schemas — the orchestrator knows every function name, argument type, and expected output. A2A exposes one coarse-grained schema per agent — the orchestrator only sends a natural language task string, and the sub-agent decides internally which of its skills to invoke. A2A is loose coupling; MCP is tight coupling.

Q2: Why does A2A use natural language task strings instead of structured function calls?

Because the sub-agent is treated as an autonomous black box. It has its own internal agent loop, its own tools (possibly MCP-based), and its own reasoning. Forcing typed args would re-introduce tight coupling. Natural language lets the orchestrator describe intent and lets the sub-agent figure out execution — which is the whole point of agent autonomy.

Q3: Why is response normalization critical, and where should it happen?

A2A responses are nested artifacts (result → artifacts → parts), while regular tool results are plain strings. The LLM only consumes ToolMessage content as strings. Normalization must happen at the wrapper layer (inside the @tool function, before returning) — not in the agent loop. This ensures the orchestrator treats local tools and A2A agents identically, with no special-casing in the graph logic.

Q4: How would you handle a sub-agent that supports streaming (streaming: true in its card)?

Instead of a single tasks/send POST waiting for a complete response, you’d use Server-Sent Events (SSE). The sub-agent streams partial artifacts back. The wrapper would accumulate SSE chunks and either stream them upstream or buffer until completion before returning a ToolMessage. The Agent Card’s capabilities.streaming flag tells you which transport mode is available.

Q5: If two sub-agents need to collaborate without going through the orchestrator, how does A2A handle that?

A2A supports peer-to-peer delegation — a sub-agent can itself act as an orchestrator and tasks/send to another agent directly. Each agent has its own Agent Card and HTTP endpoint. This enables multi-hop flows: Orchestrator → Agent A → Agent B, where Agent A is both a server (to the orchestrator) and a client (to Agent B). The orchestrator is completely unaware of this sub-delegation.

Q6: How do you prevent prompt injection when a sub-agent returns a malicious artifact content?

Treat all artifact content as untrusted user input — never interpolate it directly into system prompts or use it to construct new tool calls without sanitization. Use the normalized string only as a ToolMessage result (data plane), never as instruction (control plane). Additionally, validate artifact structure before extraction, and enforce output length limits to prevent context stuffing attacks via a crafted agent response.

Q7: What happens if a sub-agent is down during task delegation? How should the wrapper handle it?

The send_a2a_task wrapper should catch httpx.ConnectError / timeout exceptions and return a structured error string (e.g., "Agent error: Travel Agent unreachable"). The LLM receives this as a ToolMessage and can decide to retry, fall back to a different agent, or inform the user. Never let connection errors bubble up as unhandled exceptions — the agent loop must always receive a valid string result.

Q8: How is task identity managed across retries in A2A?

Each task has a client-generated UUID (task-abc-123). On retry, you can reuse the same task ID for idempotency (if the agent supports it) or generate a new one. The task ID also enables status polling via tasks/get — you can check whether a long-running task is submitted, working, completed, or failed without blocking the HTTP connection.
