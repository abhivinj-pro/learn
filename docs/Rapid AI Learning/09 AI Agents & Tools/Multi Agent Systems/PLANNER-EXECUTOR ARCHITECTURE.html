PLANNER-EXECUTOR ARCHITECTURE

# Planner Executor Architecture

# Purpose

This document consolidates the current architecture across the Planner, OnDemand Quote Orchestrator, queue-management layer, and Execution Orchestrator.

It is intended to answer four questions clearly:

  1. What each agent or layer is responsible for.
  2. How requests move between them at runtime.
  3. What payloads, events, and state objects are handed off.
  4. Why the system is built this way, where it is weak, and what future work is implied.

This document reflects the implemented architecture in the repository today. It does not describe the older future-state idea in which every agent returns a universal envelope with execution_context and external_context. That normalization remains a roadmap direction, not the live contract.

# Executive Summary

The current system is a layered quote-workflow architecture.

  • VLCOrchestrator is the front door and router.
  • OnDemandQuoteTool is the bridge from VLC into quote-specific orchestration.
  • OnDemandQuoteOrchestrator is the quote-workflow coordinator and state owner.
  • PlannerTool and PlannerOrchestrator are validation-first planning components.
  • queue tools and SimpleTaskQueue are the workflow-control mechanism for multi-step work.
  • pop_task_queue is the operational handoff into execution.
  • ExecutionOrchestrator is the task-execution agent with filtered tools.

The most important architectural truth is this:

planning and execution are not directly coupled; OnDemand and the queue layer couple them.

That design allows the system to validate first, sequence work explicitly, preserve multi-step state, and execute only the subset of tools needed for the next task group.

# System Roles

# 1. VLCOrchestrator

Role:

  • top-level orchestrator
  • classifies the incoming request at a broad capability level
  • routes quote-related requests into the OnDemand path

What it is not:

  • not the quote workflow owner
  • not the planner
  • not the executor

Practical interpretation:

VLC decides whether the request belongs to quote operations. Once it decides that, detailed quote orchestration moves down into OnDemand.

# 2. OnDemandQuoteTool

Role:

  • tool boundary used by VLC to invoke quote-specific orchestration
  • marshals the request into the OnDemand workflow
  • returns quote-agent output back to VLC

Practical interpretation:

This is the tool-level bridge between the general router and the quote workflow coordinator.

# 3. OnDemandQuoteOrchestrator

Role:

  • quote-workflow coordinator
  • owner of queue-backed workflow state
  • owner of quote-agent conversation context
  • consumer of planner output
  • aggregator of execution output

What it owns today:

  • task_queue
  • tools_additional_params
  • ondemand_conversation_context
  • final executionConversationContext
  • user-facing aggregated OnDemandServiceResponse

What it decides:

  • whether to ask Planner for decomposition or clarification
  • whether to queue work
  • whether to pop and execute work now
  • whether to respond directly with clarification or workflow status

Practical interpretation:

OnDemand is the control plane for quote workflows. It is the layer that turns a quote request into either a plan, queued work, execution, or a clarification loop.

# 4. PlannerTool and PlannerOrchestrator

Role:

  • validation-first planning agent
  • produces structured execution-ready actions
  • reports missing information explicitly

Current live output contract:

  • OutputFormat via ResponseTool

What the planner does well:

  • validates before proposing execution
  • checks schema, entity state, enums, and search results
  • sequences actions by ranking
  • emits a compact structured plan

What the planner does not do:

  • does not execute write operations
  • does not own queue insertion
  • does not return OnDemandServiceResponse

Practical interpretation:

The planner is the reasoning and validation layer that says what should happen next and with what metadata.

# 5. Queue Layer

Main components:

  • SimpleTaskQueue
  • push_back_task_queue
  • push_front_task_queue
  • pop_task_queue
  • view_task_queue
  • delete_task_queue
  • update_task_arguments

Role:

  • stores work as explicit executable queue items
  • preserves order and priority
  • supports multi-step flows across turns
  • converts planning output into an execution-control surface

Practical interpretation:

The queue layer is not a utility detail. It is the explicit workflow-control mechanism that separates deciding work from executing work.

# 6. ExecutionOrchestrator

Role:

  • executes one popped task group at a time
  • runs with a filtered authorized tool set
  • streams progress and task-status events
  • returns final execution output in the same service wrapper used by quote orchestration

Current live output contract:

  • OnDemandServiceResponse

Practical interpretation:

Execution is task-group based, tool-filtered, and invoked by queue handoff rather than by direct planner-to-executor coupling.

# Canonical Runtime Flow

User
  -> VLCOrchestrator
  -> OnDemandQuoteTool
  -> OnDemandQuoteOrchestrator
       -> PlannerTool / PlannerOrchestrator
       -> queue tools / SimpleTaskQueue
       -> pop_task_queue
       -> ExecutionOrchestrator
  -> OnDemandServiceResponse
  -> VLC final response

The control boundary changes at each step:

  • VLC owns routing.
  • OnDemand owns workflow coordination.
  • Planner owns plan generation and missing-information detection.
  • Queue tools own workflow ordering and execution handoff.
  • Executor owns actual task execution.

# Current Runtime Contracts

# 1. Planner Action Contract

# ActionFormat

{
    "action_type": "standard" | "custom" | "unsupported",
    "action_name": str,
    "metadata": dict,
    "execution_order": int,
    "original_ranking": int
}

Meaning:

  • action_name is the downstream action/tool identifier that later becomes queueable work.
  • metadata contains execution-ready arguments and dependency details.
  • execution_order is the planner-produced runtime sequence.
  • original_ranking records the ranking inherited from planner prompt rules.

# OutputFormat

{
    "executable_actions": list[ActionFormat],
    "missing_information": list[str],
    "response": str,
    "one_line_summary": str
}

Meaning:

  • executable_actions is the validated plan.
  • missing_information is what still blocks safe execution.
  • response is the user-facing planning response.
  • one_line_summary is the compact UX summary used in planner task status.

# 2. Queue Contract

# Task

{
    "name": str,
    "friendly_name": str,
    "description": str,
    "arguments": dict | None
}

# TaskWithUUID

{
    "uuid": str,
    "task": Task,
    "status": "QUEUED" | "INPROGRESS" | "COMPLETED" | "FAILED"
}

# Queue Storage Shape

Top-level queue items are either:

TaskWithUUID

or:

list[TaskWithUUID]

Meaning:

  • single item means sequential work
  • list item means a parallel task group in the same queue slot

# 3. Pop Handoff Contract

# PopTaskQueueOutput

{
    "success": bool,
    "task": TaskWithUUID | list[TaskWithUUID] | None,
    "message": str,
    "task_response": str | None,
    "queue_size": int,
    "conversationContext": list | None
}

Meaning:

  • task is the popped task group.
  • task_response is the serialized nested execution response.
  • conversationContext is the execution chat history extracted from that nested response.

# 4. Final Quote and Execution Contract

# OnDemandServiceResponse

{
    "message": str,
    "renderType": str,
    "actions": Any,
    "suggestedNextQuestions": list[str],
    "switchResource": bool,
    "conversationContext": list[ChatMessage],
    "taskQueue": str | None,
    "executionConversationContext": list[ChatMessage]
}

Important fact:

This is currently used both by the OnDemand quote path and by the Execution Orchestrator. That means execution does not yet return a dedicated normalized execution result object.

# Handoff Model

# 1. VLC to OnDemand

Handoff trigger:

  • VLC determines the request is quote-related.

Handoff mechanism:

  • VLC invokes OnDemandQuoteTool.

Handoff payload shape at a high level:

  • user query
  • pageContext
  • aad_token
  • prior quote conversation context
  • possibly persisted queue state

What changes at this boundary:

  • ownership moves from global routing to quote-specific workflow control.

# 2. OnDemand to Planner

Handoff trigger:

  • request needs decomposition, validation, sequencing, or clarification.

Handoff mechanism:

  • OnDemand invokes planner_tool.

Live planner input pattern:

{
    "query": str,
    "tools_additional_params": {
        "pageContext": dict,
        "aad_token": str,
        "ondemand_conversation_context": list
    }
}

What changes at this boundary:

  • ownership moves from workflow coordination to validation-first plan generation.

# 3. Planner back to OnDemand

Handoff trigger:

  • planner finishes and calls ResponseTool.

Handoff mechanism:

  • planner payload is emitted through FinishResponse SSE event and captured by PlannerTool.process_stream(...).

Returned payload:

  • OutputFormat

What OnDemand does with it:

  • asks for clarification if missing_information is non-empty
  • converts actions into queued tasks if execution can proceed
  • may respond immediately if the planning result itself is the desired stopping point

# 4. OnDemand to Queue

Handoff trigger:

  • planner output contains executable actions or the workflow otherwise requires explicit step management.

Handoff mechanism:

  • push_back_task_queue for normal sequencing
  • push_front_task_queue for priority insertion

What changes at this boundary:

  • work stops being a plan and becomes executable queue state.

# 5. Queue to Execution

Handoff trigger:

  • OnDemand or workflow logic calls pop_task_queue(pop_and_execute=True).

Handoff mechanism:

  • next task group is popped
  • statuses change to INPROGRESS
  • execution query is synthesized from task description and arguments
  • tool set is filtered for the popped task name or names
  • ExecutionOrchestrator.invoke(...) is called

What changes at this boundary:

  • work stops being queued intent and becomes live execution.

# 6. Execution back to OnDemand

Handoff trigger:

  • ExecutionOrchestrator finishes and emits final response.

Handoff mechanism:

  • final serialized OnDemandServiceResponse is captured in task_response
  • nested conversationContext is extracted
  • execution metadata is accumulated under execution-specific context fields

What OnDemand does with it:

  • updates user-facing result
  • updates task queue state
  • surfaces executionConversationContext
  • returns the final aggregated OnDemandServiceResponse

# Detailed Workflow by Agent

# 1. Planner Workflow

  1. OnDemand decides planning is needed.
  2. PlannerTool._arun(...) extracts pageContext, aad_token, and ondemand_conversation_context.
  3. planner emits TaskStatusEvent with INPROGRESS.
  4. PlannerOrchestrator builds its system prompt from planner prompt files.
  5. planner uses support tools such as FetchSchema, FetchEntity, FetchEnum, and search/validation tools.
  6. planner resolves what it can before claiming anything is missing.
  7. planner orders actions according to ranking rules.
  8. planner finishes with ResponseTool.
  9. final OutputFormat returns to OnDemand through the event-stream capture path.
  10. planner emits completion task status using one_line_summary.

Key property:

Planner is a validation-first action generator, not a task executor.

# 2. OnDemand Workflow

  1. receives routed quote request.
  2. rehydrates or initializes quote workflow state.
  3. decides whether to clarify, plan, queue, or execute.
  4. invokes planner when the request needs decomposition.
  5. turns executable plan items into queueable tasks.
  6. manages queue lifecycle over one or more turns.
  7. invokes pop_task_queue when work should execute.
  8. aggregates nested execution result into quote-agent response.
  9. returns unified OnDemandServiceResponse upward.

Key property:

OnDemand is the only layer that sees the full quote workflow from request intake through final response aggregation.

# 3. Queue Workflow

  1. tasks are inserted into SimpleTaskQueue.
  2. each task is wrapped with UUID and status.
  3. queue order persists execution intent.
  4. queue can hold a single sequential task or a parallel task group.
  5. view_task_queue and related tools expose and manage workflow state.
  6. pop_task_queue becomes the execution bridge when pop_and_execute=True.

Key property:

The queue is the durable operational representation of pending workflow steps.

# 4. Execution Workflow

  1. pop_task_queue pops the next task group.
  2. task status moves to INPROGRESS.
  3. execution query is synthesized from queued task content.
  4. available execution tools are filtered using queued task names.
  5. ExecutionOrchestrator runs the task group.
  6. execution streams status, debug, refresh, and response events.
  7. final OnDemandServiceResponse is emitted.
  8. nested conversationContext is extracted.
  9. queue task status is mapped to completion or failure.
  10. execution result is returned to OnDemand for final aggregation.

Key property:

Execution is controlled by queued task groups and filtered tool access, not by free-form direct invocation from Planner.

# Full End-to-End Flows

# Flow 1: Clarification-Only Flow

Use when:

  • request is incomplete or ambiguous
  • planner cannot safely generate executable actions

Path:

  1. VLC routes to OnDemand.
  2. OnDemand invokes Planner.
  3. Planner validates what it can.
  4. Planner returns missing_information and explanatory response.
  5. OnDemand returns clarification response without queue execution.

Output pattern:

  • user-facing clarification message
  • no execution yet
  • workflow may continue next turn with more information

# Flow 2: Planning-Only Flow

Use when:

  • a plan is needed first but execution is not yet performed in this turn

Path:

  1. VLC routes to OnDemand.
  2. OnDemand invokes Planner.
  3. Planner returns ordered executable_actions.
  4. OnDemand may queue the tasks but not execute them immediately, or it may stop after presenting the plan.
  5. caller receives workflow state and response.

Output pattern:

  • plan created
  • queue may be populated
  • execution deferred

# Flow 3: Plan, Queue, and Execute in One Turn

Use when:

  • request is clear enough and execution should proceed immediately

Path:

  1. VLC routes to OnDemand.
  2. OnDemand invokes Planner.
  3. Planner returns executable actions.
  4. OnDemand materializes them into Task objects.
  5. tasks are pushed into SimpleTaskQueue.
  6. OnDemand calls pop_task_queue(pop_and_execute=True).
  7. pop_task_queue invokes ExecutionOrchestrator.
  8. ExecutionOrchestrator returns nested OnDemandServiceResponse.
  9. OnDemand aggregates final result and returns upstream.

Output pattern:

  • plan created
  • queue consumed at least partially
  • execution response surfaced in same turn

# Flow 4: Multi-Turn Queue Continuation Flow

Use when:

  • workflow spans multiple dependent steps
  • user returns later
  • some work remains queued after current turn

Path:

  1. OnDemand already has existing task_queue and context from a prior turn.
  2. request re-enters the quote path.
  3. OnDemand rehydrates the queue and conversation state.
  4. OnDemand either modifies the queue, asks for clarification, or pops the next task group.
  5. execution continues from the preserved queue state.

Output pattern:

  • queue continuity across turns
  • explicit pending work visibility
  • progressive execution over time

# Flow 5: Direct Quote Handling Flow

Use when:

  • request does not need the full planner-plus-queue path
  • OnDemand can satisfy the request with direct quote-oriented behavior

Path:

  1. VLC routes to OnDemand.
  2. OnDemand handles the quote interaction directly.
  3. response is aggregated and returned without planner or executor involvement.

Output pattern:

  • simpler response path
  • lower orchestration overhead

# Event Model

The system relies on SSE-style runtime events, not just final objects.

Main event types seen across these flows:

  • ResetStatus
  • ResetResponse
  • AppendResponse
  • FinishResponse
  • TaskStatus
  • Refresh
  • Debug
  • ExecutionSteps

Why they matter:

  • they drive user-visible progress
  • they synchronize task lifecycle state
  • they transport some final payloads, especially planner output
  • they make nested orchestration observable during execution

Important nuance:

Planner’s logical contract is structured OutputFormat, but its runtime transport still depends on the event stream. Execution also streams status and response content before the final aggregated quote response exists.

# State and Context Model

# 1. Quote Workflow State

Current important state holders:

  • ondemand_conversation_context
  • task_queue
  • tools_additional_params
  • execution metadata accumulated under tools_additional_params["pop_task_queue"]

# 2. Execution Context Reality

Current execution metadata is distributed across:

  • nested tool responses
  • conversationContext
  • executionConversationContext
  • serialized queue state
  • context-store fields under tools_additional_params

This is the biggest architectural gap relative to the earlier normalized design.

# 3. Rehydration Reality

Current live behavior includes:

  • rehydrating OnDemand conversation context
  • rehydrating queue state across turns
  • accumulating execution conversation history

But there is an important limitation:

  • full execution-conversation replay back into ExecutionOrchestrator.invoke(...) is not currently active in the live pop_task_queue path.

That means execution context is preserved more as accumulated metadata than as a fully replayed deterministic execution state.

# Why This Architecture Exists

The system is gaining several concrete things from this setup.

# 1. Safer Execution

Planner validates first, which reduces blind execution attempts.

# 2. Explicit Workflow Control

The queue makes pending work visible, ordered, and manageable.

# 3. Multi-Step and Multi-Turn Support

The system can preserve partial progress instead of forcing each request into a single monolithic turn.

# 4. Separation of Concerns

  • VLC handles routing.
  • OnDemand handles quote workflow coordination.
  • Planner handles reasoning and validation.
  • Executor handles actual task execution.

# 5. Tool Surface Reduction at Execution Time

Execution runs with filtered tools per task group, which improves control and reduces accidental tool exposure.

# 6. Better UX Observability

Task status events and streaming responses give the UI richer lifecycle visibility than a simple request-response model would.

# Problems and Architectural Costs

# 1. Split Contracts

Planner returns OutputFormat, while execution and quote orchestration return OnDemandServiceResponse.

Impact:

  • more translation logic
  • harder cross-agent standardization
  • more complexity for future persistence and observability

# 2. Distributed Execution Metadata

Execution state is spread across multiple fields instead of a single deterministic execution-result object.

Impact:

  • harder downstream parsing
  • weaker guarantees for replay or analytics
  • more brittle integrations

# 3. Event-Dependent Transport

Some critical payloads are captured from streaming events instead of crossing a clean typed boundary.

Impact:

  • orchestration logic depends on transport behavior
  • debugging can be harder
  • the logical contract and transport contract are not fully separated

# 4. Planner-to-Queue Translation Is Downstream

Planner produces ActionFormat, but task materialization happens later.

Impact:

  • implicit transformation logic sits outside planner’s own contract
  • execution-readiness depends partly on downstream interpretation

# 5. Task Status Mapping Uses Heuristics

Queue-to-execution status correlation depends partly on name matching and queue matching rather than one fully stable execution identifier carried end to end.

Impact:

  • less robust lifecycle tracking
  • harder to reason about failure mapping in edge cases

# 6. Partial Execution Rehydration

Execution history is accumulated, but not fully replayed into the live execution call path.

Impact:

  • execution can lose some historical determinism
  • conversation history and execution state are not perfectly aligned

# 7. Orchestration Depth

The flow is intentionally layered, but that means more moving parts.

Impact:

  • more state transitions to reason about
  • more surfaces where contracts can drift
  • higher documentation burden

# Interview-Grade Mental Model

If asked to explain this system simply but precisely, the best framing is:

VLC routes quote work to OnDemand. OnDemand owns the quote workflow. Planner validates and proposes ordered actions. OnDemand turns those actions into queued tasks. pop_task_queue is the bridge from queued work into execution. Execution runs the next task group with filtered tools and returns an OnDemandServiceResponse. OnDemand aggregates that result, preserves queue and execution context, and returns the quote response upstream.

If asked what is unusual or important about the design, the key points are:

  • planning and execution are separated by an explicit queue layer
  • OnDemand is the workflow owner rather than Planner or Executor
  • runtime transport depends heavily on SSE events
  • current state is conversation-centric, not yet normalized into a universal execution envelope

If asked where the main architectural debt is, the answer is:

  • split contracts
  • distributed execution metadata
  • event-coupled payload transport
  • incomplete normalization of planner, quote, and executor outputs

# Future Direction

The most credible future path is not to remove the queue architecture. The queue is useful. The stronger direction is to normalize the data contracts around it.

Most likely improvements:

  1. keep Planner as a validation-first structured planner
  2. keep OnDemand as the workflow owner
  3. keep the queue as the execution-control mechanism
  4. introduce a canonical structured execution result envelope
  5. reduce dependence on mined chat history for execution metadata
  6. standardize lifecycle identifiers across planning, queueing, and execution
  7. separate transport events from logical return contracts more cleanly

This would preserve the strengths of the architecture while reducing its biggest operational ambiguities.

# Summary

This architecture is a layered quote-workflow system built around explicit handoffs.

  • VLC routes.
  • OnDemand coordinates.
  • Planner validates and plans.
  • queue tools hold and order executable work.
  • pop_task_queue hands work to execution.
  • Execution runs filtered tools and returns nested service output.
  • OnDemand aggregates the final response and preserves state for future turns.

The setup is more complex than a single-agent model, but it buys clearer role separation, safer planning, explicit multi-step workflow management, and better runtime observability. Its main costs are contract fragmentation, distributed execution state, and event-driven transport coupling.
