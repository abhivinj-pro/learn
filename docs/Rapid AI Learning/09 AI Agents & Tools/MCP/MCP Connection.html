# MCP Connection — Deep Dive

# JSON-RPC Over the Wire

MCP uses JSON-RPC 2.0 as its message format — a lightweight spec where every message is structured as:

// Request (client → server)
{
  "jsonrpc": "2.0",
  "id": "abc-123",          // used to correlate response
  "method": "tools/call",
  "params": {
    "name": "db_query",
    "arguments": { "user_ids": [12, 45] }
  }
}

// Response (server → client)
{
  "jsonrpc": "2.0",
  "id": "abc-123",          // same id — client matches this to the request
  "result": {
    "content": [{ "type": "text", "text": "..." }],
    "isError": false
  }
}

JSON-RPC itself says nothing about transport — it’s just a message envelope. MCP layers it on top of stdio or SSE.

# MCP vs REST — The Core Difference

Dimension REST API MCP (SSE transport)
Connection New TCP connection per request One persistent HTTP connection, kept open
Direction Client pulls only Server can push messages to client without a request
Protocol HTTP request/response HTTP + SSE stream (uni-directional push) + HTTP POST (client sends)
Session state Stateless by design Stateful — server tracks the session
Multiplexing Each call is independent Multiple JSON-RPC calls in-flight over same connection, matched by id

With REST, every GET /tools/call opens a socket, gets a response, closes. With MCP over SSE:

sequenceDiagram
  participant C as Client
  participant S as MCP Server
  C->>S: GET /sse
  S-->>C: 200 OK, Content-Type text/event-stream
  S-->>C: event endpoint, data /messages
  Note over C,S: connection stays open
  C->>S: POST /messages (tools/list)
  S-->>C: event message, data {result...}
  C->>S: POST /messages (tools/call)
  S-->>C: event message, data {result...}

Key insight: The client sends via HTTP POST, but receives via the already-open SSE stream. Two channels, one logical session. This is why the async with block must stay open — closing it tears down the SSE stream, so there’s nowhere to receive tools/call results.

# stdio Transport — Even Simpler

When using stdio, there’s no HTTP at all:

flowchart LR
  o["Orchestrator Process"] -->|"spawns subprocess"| m["MCP Server Process"]
  o -->|"writes JSON-RPC to stdin"| m
  m -->|"reads JSON-RPC from stdout"| o

Fully synchronous, same machine, zero network. The “connection” is literally two OS pipes. This is why it’s only viable for local tools — you can’t share it across processes or machines.

# How Multiple Clients Connect to the Same SSE Server

This is where it gets important for production. Each client opening GET /sse gets its own isolated SSE stream — the server assigns a unique session ID per connection:

flowchart LR
  a["Client A"] -->|"GET /sse"| s1["Server · session s1"]
  b["Client B"] -->|"GET /sse"| s2["Server · session s2"]
  c["Client C"] -->|"GET /sse"| s3["Server · session s3"]
  s1 -.->|"tools/call results"| a
  s2 -.->|"tools/call results"| b
  s3 -.->|"tools/call results"| c

When Client A does POST /messages, it includes its session ID so the server routes the response back to Client A’s SSE stream only. Sessions are fully isolated — one client’s tool calls don’t bleed into another’s.

Concurrency model on the server side depends on implementation:

  • Single-process async (e.g., FastAPI + asyncio): handles many SSE connections concurrently in one process — works well for I/O-bound tools
  • Multi-process/threaded: needed if tools are CPU-bound or block the event loop
  • Stateless server with external session store: session state (active tools, auth context) stored in Redis/DB — allows horizontal scaling across multiple server replicas

# Why This Matters for Agent Design

# This is NOT just "calling an API"
# It's maintaining a stateful session where:
#   - the server knows WHO you are (session)
#   - in-flight requests are multiplexed (matched by JSON-RPC id)
#   - the server can theoretically push unsolicited events (notifications)

# MCP spec defines server→client notifications too, e.g.:
# notifications/tools/list_changed  ← server tells client its tools changed
# notifications/resources/updated   ← server pushes a resource update

This notification capability is something REST literally cannot do without polling or webhooks — the persistent connection earns its cost here. A future-proof MCP client should handle notifications/tools/list_changed to dynamically re-bind tools mid-session without a reconnect.
