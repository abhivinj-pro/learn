# Tool Retrieval Overview

# High-Level Purpose

This code selects the most relevant tools from a toolbox based on what the user is asking for. Think of it like a smart assistant that, given a user’s question, figures out which tools or functions are most useful to answer it.

Referenced file:

  • CustomToolRetriever.py

# Classes

# 1. ReInvoke_ToolRetriever (Abstract Base Class)

This is a template class that:

  • Stores an LLM (self.model) and a system prompt.
  • Defines get_intent(), which extracts what the user wants from chat history.
  • Declares an abstract retrieve() method that child classes must implement.

# 2. MaxSim_CrossEncoder_ToolRetriever (Concrete Implementation)

This is the main class that implements the actual tool retrieval logic using a Cross-Encoder model.

# Walkthrough with Example

Assume the following:

  • User’s question: “Can you find the price of product X and also check inventory?”
  • Available tools: [PriceLookupTool, InventoryTool, ShippingTool, CustomerSupportTool, ReportTool]
  • top_k = 3, meaning we want the top 3 most relevant tools.

# Step 1: Initialization (__init__)

tool_retriever = MaxSim_CrossEncoder_ToolRetriever(
	model=llm_model,
	system_prompt="You are a helpful assistant...",
	dependency_dict={"PriceLookupTool": ["InventoryTool"]},  # dependencies
	cross_encoder_model_name="cross-encoder/ms-marco-MiniLM-L-6-v2",
	local=True,
)

What happens:

  1. Stores the LLM model and system prompt inherited from the parent.
  2. Stores a dependency_dict. If tool A needs tool B to work, this tracks that dependency.
  3. Loads the Cross-Encoder model only once and caches it in _initialized_models.

# ML Concept: What Is a Cross-Encoder?

A Cross-Encoder is a type of neural network model, usually based on BERT or Transformers, that:

  • Takes two texts as input simultaneously, for example a query and a document.
  • Outputs a single relevance score between them.

Why Cross-Encoder vs. Bi-Encoder?

  • Bi-Encoder: encodes each text separately, then compares embeddings. Fast, but less accurate.
  • Cross-Encoder: encodes both texts together, allowing deep interaction between them. Slower, but more accurate for ranking tasks.

The model ms-marco-MiniLM-L-6-v2 was trained on the MS MARCO dataset, a large-scale question-answering dataset from Microsoft, to predict how relevant a passage is to a query.

# Step 3: Intent Extraction (get_intent)

intent = self.get_intent(chat_history)

The code takes the chat history, separates the current message from previous messages, and asks the LLM:

Given this conversation, what are the user’s intents for tool usage?

Example output:

1. Look up the price of product X
2. Check inventory levels

The intent is then split by newlines into sub_intents:

sub_intents = [
	"1. Look up the price of product X",
	"2. Check inventory levels",
]

Why extract intents?

A single user message might require multiple tools. Breaking it into sub-intents allows the system to match each intent to the best tool.

# Step 4: Building Query-Tool Pairs

pairs = []

for sub_intent in sub_intents:
	pairs.extend([
		(sub_intent, tool.documentation or tool.description)
		for tool in tools
	])

For this example, the code creates:

pairs = [
	("1. Look up the price of product X", "PriceLookupTool: Finds product prices..."),
	("1. Look up the price of product X", "InventoryTool: Checks stock levels..."),
	("1. Look up the price of product X", "ShippingTool: Calculates shipping..."),
	("1. Look up the price of product X", "CustomerSupportTool: Handles support..."),
	("1. Look up the price of product X", "ReportTool: Generates reports..."),
	("2. Check inventory levels", "PriceLookupTool: Finds product prices..."),
	("2. Check inventory levels", "InventoryTool: Checks stock levels..."),
	...,
]

Total pairs:

num_sub_intents × num_tools = 2 × 5 = 10

# Step 5: Cross-Encoder Scoring

if self.local:
	all_scores = cross_encoder.predict(pairs)
else:
	all_scores = self.request(pairs, aad_token=aad_token)

The Cross-Encoder processes all pairs and returns a relevance score for each:

all_scores = [
	0.92, 0.15, 0.05, 0.10, 0.08,  # Scores for intent 1 vs all tools
	0.12, 0.95, 0.06, 0.11, 0.07,  # Scores for intent 2 vs all tools
]

Higher score means more relevant.

# ML Concept: How Cross-Encoder Works Internally

  1. The two texts are concatenated with special tokens: [CLS] intent [SEP] tool_description [SEP].
  2. This goes through a Transformer, typically a BERT-like architecture.
  3. The [CLS] token output is passed through a linear layer to get a single score.
  4. The model is trained to output high scores for relevant pairs and low scores for irrelevant ones.

# Step 6: Softmax Normalization

sub_intent_tool_scores = softmax(sub_intent_tool_scores)

The softmax function at the top of the file:

def softmax(x):
	e_x = np.exp(x - np.max(x))  # Subtract max for numerical stability
	return e_x / e_x.sum(axis=0)

# ML Concept: What Is Softmax?

Softmax converts raw scores into probabilities that sum to 1:

softmax(xi​)=∑j​exj​exi​​

Example:

  • Raw scores: [0.92, 0.15, 0.05, 0.10, 0.08]
  • After softmax: [0.52, 0.24, 0.11, 0.08, 0.05] approximately

Why subtract max?

For numerical stability. Without it, exp() of large numbers can overflow.

Why use softmax here?

It normalizes scores so they are comparable across different sub-intents and emphasizes the differences between tools.

# Step 7: Tool Selection Logic

For each sub-intent:

max_score_index = np.argmax(sub_intent_tool_scores)

if tools[max_score_index].name not in relevant_tools:
	relevant_tools.append(tools[max_score_index].name)

sub_intent_tool_scores[max_score_index] -= 100.0  # Effectively remove from consideration

What is happening:

  1. Find the tool with the highest score for this sub-intent.
  2. Add it to relevant_tools if it is not already present.
  3. Subtract 100 from its score as a simple way to remove it from later max selection.

For this example:

  • Intent 1 -> best tool: PriceLookupTool
  • Intent 2 -> best tool: InventoryTool
relevant_tools = ["PriceLookupTool", "InventoryTool"]

# Step 8: Track Maximum Scores per Tool

for tool_index, tool in enumerate(tools):
	tool_name = tools[tool_index].name
	final_tool_scores[tool_name] = max(
		final_tool_scores[tool_name],
		sub_intent_tool_scores[tool_index],
	)

This keeps track of the best score each tool received across all sub-intents. This is the “MaxSim” or maximum similarity part of the class name.

# ML Concept: Why Max Similarity?

When a tool is relevant to any sub-intent, that should count. Taking the maximum instead of the average ensures that a tool that is highly relevant to one intent is not penalized for being irrelevant to others.

# Step 9: Fill Remaining Slots

sorted_tools = sorted(final_tool_scores.items(), key=lambda x: x[1], reverse=True)

for tool in sorted_tools:
	if len(relevant_tools) >= top_k:
		break
	if tool[0] not in relevant_tools:
		relevant_tools.append(tool[0])

If the list has not reached top_k yet, the remaining slots are filled with the next highest-scoring tools.

Example:

If top_k = 3 and you currently have 2 tools, the code adds the next best tool, perhaps CustomerSupportTool.

# Step 10: Add Dependencies

for tool in relevant_tools:
	for dep_tool in self.dependency_dict.get(tool, []):
		if dep_tool not in relevant_tools and dep_tool in [t.name for t in tools]:
			relevant_tools.append(dep_tool)

If PriceLookupTool depends on InventoryTool through dependency_dict, this step ensures InventoryTool is included.

# Final Output

return relevant_tools

# Returns: ["PriceLookupTool", "InventoryTool", "CustomerSupportTool"]

# Summary Flow Diagram

flowchart TD
	A[User Query]
	B[LLM extracts sub-intents<br/>Price lookup + Inventory]
	C[Create intent-tool pairs for all combinations]
	D[Cross-Encoder scores each pair for relevance]
	E[Softmax normalizes scores per sub-intent]
	F[Select best tool per intent and fill remaining top_k slots]
	G[Add dependent tools]
	H[Return selected tools]

	A --> B --> C --> D --> E --> F --> G --> H

# Why This Architecture?

  1. Two-stage approach: the LLM understands complex language and breaks down intents, while the Cross-Encoder ranks tools for those intents.
  2. Cross-Encoder for ranking: this is more accurate than basic embedding similarity for tool matching.
  3. Thread-safe model caching: _lock and _initialized_models avoid loading expensive ML models more than once in multi-threaded environments.
  4. Local vs. remote execution: the Cross-Encoder can run locally for lower latency or through an API when local inference is not practical.
