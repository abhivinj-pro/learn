Async Programming - Python

A friendly, experiment-driven guide to async/await. Every idea comes with a tiny program you can actually run — watch the output, tweak it, break it, and the concept sticks for good. No memorizing; just seeing.

How to use this guide. Keep a terminal open. Every 🧪 Experiment is a complete, copy-paste-runnable file — save it as x.py and run python x.py (Python 3.11+). The predicted output is shown right below each one. The real learning happens when you do the Try this tweaks and check whether reality matches your prediction.

1. The Core Mental Model

One sentence to anchor everything: Async Python is one worker who is very good at never standing around waiting. While one job is stuck waiting (for the network, a file, a timer), the worker quietly goes and makes progress on another job — but it only ever does one thing at a time.

The coffee-shop barista (your mental picture)

Picture a single barista. A customer orders a pour-over that needs 3 minutes to drip. Two ways to run the shop:

  • Synchronous barista: starts the pour-over, then stares at it dripping for 3 minutes while 10 people queue. Useless waiting.
  • Async barista: starts the pour-over, and the moment it's "just dripping on its own" (a waiting step), turns to take the next order, start a second drip, ring up a muffin… When a drip finishes, they come back and hand it over.

Same one barista. Same two hands. No clones. They just refuse to stand idle. That is async: not more workers, just zero wasted waiting.

Let's prove it — your first experiment

Below, boil_water "takes" 2 seconds. We do it three times. Run the sync version, then the async version, and look at the wall-clock time.

🧪 Experiment 1 — the same work, two speeds
import asyncio, time

async def boil_water(name):
    print(f"{name}: put kettle on")
    await asyncio.sleep(2)          # the "waiting" step — water boils on its own
    print(f"{name}: {name} is ready!")

# --- Version A: one after another (we await immediately) ---
async def sequential():
    start = time.perf_counter()
    await boil_water("kettle-1")
    await boil_water("kettle-2")
    await boil_water("kettle-3")
    print(f"sequential took {time.perf_counter() - start:.1f}s\n")

# --- Version B: all three kettles at once ---
async def concurrent():
    start = time.perf_counter()
    await asyncio.gather(
        boil_water("kettle-1"),
        boil_water("kettle-2"),
        boil_water("kettle-3"),
    )
    print(f"concurrent took {time.perf_counter() - start:.1f}s")

asyncio.run(sequential())
asyncio.run(concurrent())
Predicted output

kettle-1: kettle-1 is ready!
kettle-2: put kettle on
kettle-2: kettle-2 is ready!
kettle-3: put kettle on
kettle-3: kettle-3 is ready!
sequential took 6.0s

kettle-2: put kettle on
kettle-3: put kettle on
kettle-1: kettle-1 is ready!

Look closely at the order of the lines. In the sequential run, each kettle fully finishes before the next starts. In the concurrent run, all three "put kettle on" lines print first — the barista started all three drips — and only then do the "ready" lines come back. Three 2-second waits overlapped into a single 2 seconds.

The click: Nothing ran faster. asyncio.sleep(2) still took 2 seconds every time. We just stopped doing those waits one at a time. Async didn't add speed — it removed idle waiting.
Try this and predict first:
  • Change asyncio.sleep(2) to time.sleep(2) (the blocking, non-async version) in boil_water and re-run the concurrent version. What happens to the time? (Spoiler explored in §11 — this is the single most important gotcha.)
  • Add a 4th and 5th kettle to concurrent(). Does the total time change?

The four intuitions everything else is built on

  1. Async is "better waiting," not "faster doing." The win is purely from overlapping idle time. If there's no waiting, async does nothing for you.
  2. You switch away, you're never yanked away. Control only changes hands at an await. Between two awaits, your code has the whole stage to itself — guaranteed.
  3. Calling an async function doesn't run it. It hands you a "coroutine object" — a job ticket. Someone has to await it or schedule it for anything to happen (we'll see this in §5).
  4. Async is contagious. To await something you must be inside an async def. So async tends to spread up the call stack — the "function color" idea.

2. Concurrency vs Parallelism vs Async

People throw these three words around as if they're the same. They're not, and once you feel the difference you'll never confuse them again. Here's the plain-English version:

  • Concurrency = juggling. One juggler, several balls in the air. At any instant exactly one ball is in the hand, but all of them are "in progress." It's about structure — dealing with many things at once.
  • Parallelism = several jugglers, each with their own ball, all moving at the same physical instant. It's about hardware — doing many things at once on many CPU cores.
  • Async = a specific juggling technique: one juggler who only switches balls at the exact moment a ball is mid-air (waiting). Cooperative, single-threaded concurrency.

Rob Pike's famous line nails it: "Concurrency is dealing with lots of things at once. Parallelism is doing lots of things at once." Async is a way to deal with, never a way to do.

Feel the difference: when does async actually help?

This is the experiment that settles every argument. We run two kinds of "work": one that waits (I/O-bound, simulated with asyncio.sleep) and one that computes (CPU-bound, a real number crunch). Watch which one async speeds up.

🧪 Experiment 2 — async helps waiting, not computing
import asyncio, time

# I/O-bound: mostly waiting
async def io_task(n):
    await asyncio.sleep(1)      # pretend: a network call
    return n

# CPU-bound: mostly computing (no awaiting anywhere)
async def cpu_task(n):
    total = 0
    for i in range(10_000_000):
        total += i * i
    return n

async def run(label, task):
    start = time.perf_counter()
    await asyncio.gather(*(task(i) for i in range(4)))
    print(f"{label}: {time.perf_counter() - start:.2f}s")

async def main():
    await run("4x IO  (await sleep)", io_task)   # 4 one-second waits
    await run("4x CPU (number crunch)", cpu_task) # 4 heavy loops

asyncio.run(main())
Typical output
4x IO  (await sleep):   1.00s   ← 4 seconds of waiting collapsed into 1
4x CPU (number crunch): 2.80s   ← no speedup; they ran one-after-another anyway
The click: The 4 I/O tasks overlapped (4s → 1s) because each one paused at await and let the others run. The 4 CPU tasks did not overlap — they never hit an await, so the single worker ground through them back-to-back. For CPU work you need real parallelism (processes), which we cover in §11.
Try this: swap the CPU loop for multiprocessing or wrap it in asyncio.to_thread (§11) and watch whether the CPU number drops. It won't drop with threads (the GIL!), but will with processes.

Reach for async when…

  • You're waiting on the network — LLM APIs, vector DBs, HTTP, gRPC.
  • You need thousands of live connections (WebSocket, SSE).
  • You want simple, predictable interleaving with few locks.

Reach for processes/threads when…

  • CPU-bound: tokenizing, parsing megabytes, image preprocessing → multiprocessing.
  • You're stuck with a blocking library → wrap it in asyncio.to_thread.

3. The Event Loop, Demystified

"Event loop" sounds mystical. It isn't. It's a to-do list manager running in a while True. It keeps a list of jobs that are "ready to run right now" and a list of jobs that are "parked, waiting for something." On every spin it does three boringly simple things:

while there_is_work:
    # 1. Run every job that is READY, one at a time, until each one
    #    either finishes or hits an `await` and parks itself.
    run_all_ready_jobs()

    # 2. Ask the operating system, in ONE syscall:
    #    "Out of these 10,000 sockets/timers I'm watching, which are done?"
    ready_now = os_wait_for_any(parked_jobs)   # epoll / kqueue / IOCP

    # 3. Move those finished ones from "parked" back to "ready".
    wake_up(ready_now)

That step 2 is the whole trick. The OS gives you one call that means "sleep until any of these thousands of things is ready." So a single worker can babysit tens of thousands of connections without polling each one.

Watch the loop juggle (see the interleaving with your own eyes)

Each task announces every step it takes. The await asyncio.sleep(...) is the moment a task says "I'm parking myself — loop, go run someone else."

🧪 Experiment 3 — three tasks taking turns
import asyncio

async def task(name, naps):
    for i in range(naps):
        print(f"  {name} step {i}  (now I'll await → loop can switch)")
        await asyncio.sleep(0.1)   # park here; let others run
    print(f"  {name} DONE")

async def main():
    await asyncio.gather(
        task("A", 3),
        task("B", 2),
        task("C", 1),
    )

asyncio.run(main())
Predicted output (note the round-robin)
  A step 0  (now I'll await → loop can switch)
  B step 0  (now I'll await → loop can switch)
  C step 0  (now I'll await → loop can switch)
  A step 1  (now I'll await → loop can switch)
  B step 1  (now I'll await → loop can switch)
  C DONE
  A step 2  (now I'll await → loop can switch)
  B DONE
  A DONE

Read it top to bottom: A runs one step and parks, B runs one step and parks, C runs one step and parks, then the loop comes back around to A. They take turns at the await points. C finishes first (only 1 nap), then B, then A. This round-robin is the event loop doing its job.

The click: The event loop never "interrupts" a task. Each task runs uninterrupted until it voluntarily hits await and parks itself. The loop simply picks whoever is ready next. Cooperative, not forced.
Try this: Replace await asyncio.sleep(0.1) with nothing (delete that line). Now each task runs all its steps before yielding — A fully finishes, then B, then C. No await = no turn-taking. That one change is the entire lesson.

Why this design is so cheap

  • Switching tasks is almost free — it's just the loop picking the next item off a list (~100 nanoseconds). Switching OS threads costs 10–100× more.
  • You rarely need locks — between two awaits nobody else can touch your data. (More in §10.)
  • Tasks are tiny — a parked coroutine is ~1 KB of memory; an OS thread reserves megabytes of stack.
The one rule that matters: the loop runs in your single thread. If a task never reaches await — or calls a blocking function like time.sleep(), requests.get(), or a giant json.loads() — it hogs the worker and every other task freezes. This is the #1 cause of "why is my async app slow?" We'll see it live in §11.

4. Generators — the Engine Under Coroutines

This is the missing puzzle piece. Once generators click, coroutines stop feeling magical — because a coroutine literally is a generator with a few extra powers.

Here's a question that trips everyone up: why does calling an async function return a "coroutine object" instead of just running and giving you an answer? The honest answer is: because async borrowed that behavior from generators, a feature Python has had since 2001. So let's master the simple thing first, and the async thing will fall into place.

What a generator actually is

A normal function uses return and hands back one value, then forgets everything. A generator is a function that uses yield instead. The magic of yield: it hands back a value and pauses the function in place, remembering every local variable, ready to continue from that exact spot next time you ask. It's a function that can be paused and resumed.

The catch that confuses people: calling a generator function runs none of its body. It just builds and hands you a "generator object" — a remote control. The code only runs when you drive the remote control by iterating.

🧪 Experiment 4 — calling it runs nothing; you must drive it
def count_up_to(n):
    print("  >> generator body STARTS now")
    i = 1
    while i <= n:
        print(f"  >> about to yield {i}")
        yield i                       # pause here, hand out i
        print(f"  >> resumed after yielding {i}")
        i += 1
    print("  >> generator body ENDS")

print("1. calling count_up_to(3) ...")
gen = count_up_to(3)                  # builds a generator — runs NOTHING
print(f"2. got a {type(gen).__name__}; no body output above this line")

print("3. calling next(gen) the first time:")
print("   ->", next(gen))             # runs up to the first yield
print("4. calling next(gen) again:")
print("   ->", next(gen))             # resumes, runs to next yield
Predicted output
1. calling count_up_to(3) ...
2. got a generator; no body output above this line
3. calling next(gen) the first time:
  >> generator body STARTS now
  >> about to yield 1
   -> 1
4. calling next(gen) again:
  >> resumed after yielding 1
  >> about to yield 2
   -> 2
The click: Look at step 2 — calling count_up_to(3) printed nothing from the body. The function didn't run; it became a paused object. Each next() presses "play" until the next yield, then pauses again. That pause-and-remember is exactly what await does for coroutines.

Why you "have to do for i in gen()"

You noticed that the result doesn't appear until you write for i in something(). Now you know why: the for loop is just calling next() for you over and over until the generator is exhausted. for is the "press play repeatedly until it's done" button.

🧪 Experiment 5 — for is just automatic next()
def squares(n):
    for i in range(1, n + 1):
        yield i * i

gen = squares(5)
print("the object:", gen)            # <generator object ...> — not the values!

# Option A: drive it with a for-loop (the usual way)
for value in squares(5):
    print("got", value)

# Option B: force ALL values at once by materializing into a list
print("as a list:", list(squares(5)))
Predicted output
the object: <generator object squares at 0x...>
got 1
got 4
got 9
got 16
got 25
as a list: [1, 4, 9, 16, 25]

Printing the generator object shows <generator object ...> — proof it's a remote control, not the values. To get values you either loop over it, call next() by hand, or slurp them all with list().

Why generators are worth having: laziness

Because a generator computes values on demand, one at a time, it never has to hold them all in memory. That lets you represent huge — even infinite — sequences for free, and stop early without wasting work.

🧪 Experiment 6 — an infinite sequence that costs almost no memory
def naturals():            # infinite! this would be impossible with a list
    i = 0
    while True:
        yield i
        i += 1

gen = naturals()
# Take just the first 5 — the generator only computes those 5, then pauses forever.
first_five = [next(gen) for _ in range(5)]
print(first_five)

# A list version, list(range(10_000_000)), would allocate 10M ints up front.
# The generator holds ONE int at a time. That's the whole point of laziness.
Predicted output
[0, 1, 2, 3, 4]
Try this: replace [next(gen) for _ in range(5)] with list(naturals()) and run it. It hangs forever (or crashes on memory) because list() tries to pull every value from an infinite generator. That contrast is the clearest demonstration of lazy vs eager.

The bridge: a coroutine is a generator the event loop drives

Now the payoff. Compare the two side by side — they are the same idea:

GeneratorCoroutine (async)
You define it withdef + yieldasync def + await
Calling it returnsa generator object (nothing runs)a coroutine object (nothing runs)
It pauses atyieldawait
Who presses "play"you, via for / next()the event loop, via await / asyncio.run

That's not an analogy — it's the actual implementation. Under the hood await is a type-checked yield, and the event loop is the "for-loop" that keeps pressing play on thousands of paused coroutines. When you saw in §1 that greet("A") printed nothing until awaited, that's identical to count_up_to(3) printing nothing until you call next().

Untangle the three "for/await" forms — this is the exact spot people mix up:
  • Generator → for x in gen():  (synchronous, you drive it)
  • Coroutine → await coro()  (you do not loop over it; you await the single result)
  • Async generator → async for x in agen():  (a generator that also awaits between yields — covered in §12)
So "for i in coroutine()" is actually a mistake — that's the generator pattern. A plain coroutine is awaited; only an async generator is looped over, and with async for.
The one-sentence takeaway: a generator is a pausable function you drive with next()/for; a coroutine is the same pausable function, but the event loop drives it and the pause keyword is await. Calling either one builds an object and runs nothing — something has to drive it.

5. Coroutines, Tasks, and Futures

Three words that sound interchangeable but aren't. The simplest way to keep them straight:

  • A coroutine is a job ticket. You get one by calling an async def function. Holding a ticket does nothing — the work hasn't started.
  • A Task is a job ticket you handed to the loop and said "start working on this in the background." It runs alongside everything else.
  • A Future is a low-level "I promise a value will show up here later" box. A Task is just a fancy Future. You rarely make one yourself.

Prove it: calling an async function runs nothing

This surprises everyone the first time. Calling greet() below does not print anything — it just builds a coroutine object. Only await (or scheduling) makes it run.

🧪 Experiment 7 — the ticket vs. the work
import asyncio

async def greet(name):
    print(f"Hello, {name}!")
    return f"done-{name}"

async def main():
    print("1. calling greet('A') ...")
    ticket = greet("A")          # builds a coroutine — prints NOTHING yet
    print(f"2. got a {type(ticket).__name__}; still no greeting printed")

    print("3. now awaiting it ...")
    result = await ticket        # NOW greet actually runs
    print(f"4. it returned: {result}")

asyncio.run(main())
Predicted output
1. calling greet('A') ...
2. got a coroutine; still no greeting printed
3. now awaiting it ...
Hello, A!
4. it returned: done-A
The click: greet("A") is like writing an order on a slip — the kitchen hasn't seen it. await (or create_task) is handing the slip in. If you ever see RuntimeWarning: coroutine 'greet' was never awaited, it means you made a ticket and threw it in the bin without handing it in.

The difference that makes or breaks performance

This is the single most common "I added async and nothing got faster" bug. await coro runs it now, inline (you wait for it before moving on). create_task(coro) starts it in the background so several can run at once.

🧪 Experiment 8 — inline await vs. background task
import asyncio, time

async def work(n):
    await asyncio.sleep(1)
    return n * 10

async def the_slow_way():           # awaiting immediately = sequential
    start = time.perf_counter()
    results = []
    for n in range(3):
        results.append(await work(n))   # wait fully before next loop
    print(f"slow: {results} in {time.perf_counter()-start:.1f}s")

async def the_fast_way():           # schedule all, THEN await = concurrent
    start = time.perf_counter()
    tasks = [asyncio.create_task(work(n)) for n in range(3)]  # all start now
    results = [await t for t in tasks]                        # collect later
    print(f"fast: {results} in {time.perf_counter()-start:.1f}s")

async def main():
    await the_slow_way()
    await the_fast_way()

asyncio.run(main())
Predicted output
slow: [0, 10, 20] in 3.0s
fast: [0, 10, 20] in 1.0s

Same results, same code shape — but the fast way create_tasks all three before awaiting any of them, so the three 1-second waits overlap. The slow way awaits inside the loop, finishing each before starting the next.

Try this: In the_fast_way, change the two lines to a single results = await asyncio.gather(*(work(n) for n in range(3))). Same 1-second result — gather is just the shortcut for "schedule all, then wait for all."

6. What await Actually Does

If async has ever felt like a magic word, this section dissolves the magic. await does exactly one conceptual thing: it's a "pause point" where your function says to the loop, "I'm about to wait — go do something useful and wake me when my thing is ready."

Step by step, when Python reaches await something:

  1. It looks at something (must be "awaitable" — a coroutine, Task, or Future).
  2. It freezes your function exactly where it is — local variables, the line you're on, everything — and stores that frozen state.
  3. It hands control back to the event loop with a note: "wake me when something finishes."
  4. The loop is now free to run other ready tasks.
  5. When something is done, the loop thaws your function and continues from right after the await, with the result dropped in where await something was.

See the freeze-and-thaw happen

Two tasks, each printing before and after its await. The interleaving proves that await is where one function freezes and another gets a turn.

🧪 Experiment 9 — freeze here, resume there
import asyncio

async def job(name):
    print(f"{name}: BEFORE await (running)")
    await asyncio.sleep(0)        # freeze point — hand control to the loop
    print(f"{name}: AFTER await (resumed)")

async def main():
    await asyncio.gather(job("X"), job("Y"))

asyncio.run(main())
Predicted output
X: BEFORE await (running)
Y: BEFORE await (running)
X: AFTER await (resumed)
Y: AFTER await (resumed)

Notice both "BEFORE" lines print before either "AFTER." X runs up to its await and freezes; Y then runs up to its await and freezes; then the loop thaws X, then Y. The await is the seam where control changes hands.

The click: await asyncio.sleep(0) waits for zero seconds — so why use it? Because the act of awaiting is itself the yield. It means "I have nothing to wait for, but I'll politely give someone else a turn." It's the cleanest way to see (and create) a switch point.

The flip side: no await means no one else runs

Because switching only happens at await, a function with no await in its hot path monopolizes the worker. This is the trap that makes "async" code behave synchronously.

🧪 Experiment 10 — the task that never yields
import asyncio

async def hog():
    print("hog: starting a long loop with NO await")
    total = 0
    for i in range(50_000_000):   # heavy, and never awaits
        total += i
    print("hog: finally done")

async def chatty():
    for i in range(5):
        print(f"chatty: tick {i}")
        await asyncio.sleep(0.2)

async def main():
    await asyncio.gather(hog(), chatty())

asyncio.run(main())
Predicted output — chatty is frozen out until hog finishes
hog: starting a long loop with NO await
hog: finally done
chatty: tick 0
chatty: tick 1
chatty: tick 2
chatty: tick 3
chatty: tick 4

chatty wanted to tick every 0.2s, but it couldn't print a single tick until hog completely finished — because hog never reached an await to release the worker.

Try this: add await asyncio.sleep(0) inside hog's loop (say, every million iterations: if i % 1_000_000 == 0: await asyncio.sleep(0)). Re-run — now chatty's ticks interleave with the loop. You just made a greedy task cooperative. That's the whole idea of cooperative multitasking.

Under the hood (nice to know, not need to know): coroutines are built on Python's generator machinery — await is essentially a type-checked yield from. That's why async stack traces look like nested dolls: each frozen frame is a paused generator waiting to resume.

7. Core Concurrency Patterns

These five tools cover ~90% of real async code. Each one answers a different "I have many things to run — how do I coordinate them?" question.

7.1 gather — "run all of these, give me all the results"

The workhorse. Schedules everything at once, waits for all, returns results in the order you passed them (not the order they finished).

🧪 Experiment 11 — gather keeps input order
import asyncio

async def fetch(name, delay):
    await asyncio.sleep(delay)
    print(f"  {name} finished (took {delay}s)")
    return name.upper()

async def main():
    results = await asyncio.gather(
        fetch("slow", 0.3),
        fetch("fast", 0.1),
        fetch("medium", 0.2),
    )
    print("results:", results)   # order matches the call order, not finish order

asyncio.run(main())
Predicted output
  fast finished (took 0.1s)
  medium finished (took 0.2s)
  slow finished (took 0.3s)
results: ['SLOW', 'FAST', 'MEDIUM']
The famous gather footgun: if one coroutine raises, gather re-raises that error immediately — but the other tasks keep running in the background, now untracked. Pass return_exceptions=True to instead collect errors as values: await asyncio.gather(*coros, return_exceptions=True) returns a mix of results and exception objects, nothing is lost. (For automatic cancel-the-siblings behavior, use TaskGroup in §8.)

7.2 as_completed — "give me results the instant each one is ready"

Same fan-out, but you process each result as it arrives instead of waiting for the slowest. Perfect for streaming progress to a user.

🧪 Experiment 12 — fastest-first delivery
import asyncio

async def fetch(name, delay):
    await asyncio.sleep(delay)
    return f"{name} (after {delay}s)"

async def main():
    coros = [fetch("slow", 0.3), fetch("fast", 0.1), fetch("medium", 0.2)]
    for coro in asyncio.as_completed(coros):
        result = await coro      # arrives in COMPLETION order
        print("got:", result)

asyncio.run(main())
Predicted output — fastest first
got: fast (after 0.1s)
got: medium (after 0.2s)
got: slow (after 0.3s)

Compare with Experiment 11: same tasks, but here you act on each result the moment it lands. Great for "show me search hits as they come back."

7.3 wait_for — "but give up after N seconds"

Wraps a single awaitable with a deadline. If it doesn't finish in time, it's cancelled and you get a TimeoutError.

🧪 Experiment 13 — a timeout with a fallback
import asyncio

async def slow_api():
    await asyncio.sleep(5)
    return "real answer"

async def main():
    try:
        answer = await asyncio.wait_for(slow_api(), timeout=1.0)
    except asyncio.TimeoutError:
        answer = "fallback (api too slow)"
    print(answer)

asyncio.run(main())
Predicted output (after ~1 second)
fallback (api too slow)

7.4 wait — "wake me when the first one finishes (or on any condition)"

done, pending = await asyncio.wait(
    tasks,
    timeout=5,
    return_when=asyncio.FIRST_COMPLETED,   # or FIRST_EXCEPTION, ALL_COMPLETED
)
for p in pending:
    p.cancel()                              # tidy up the losers

Lower-level than gather: it hands you two sets — what finished and what's still pending — and lets you decide what to do next. The classic use is "race several options, take the first, cancel the rest" (see §13.3).

7.5 Queue — "producers and consumers at different speeds"

A thread-safe-for-the-loop mailbox. Producers put items in; consumers get them out. A bounded queue (maxsize) gives you backpressure for free: if consumers fall behind, put politely blocks so producers slow down instead of blowing up memory.

🧪 Experiment 14 — one producer, two consumers
import asyncio

async def producer(q, n):
    for i in range(n):
        await q.put(i)
        print(f"produced {i}")
    await q.put(None); await q.put(None)   # one sentinel per consumer

async def consumer(q, name):
    while (item := await q.get()) is not None:
        await asyncio.sleep(0.1)           # pretend processing
        print(f"  {name} handled {item}")

async def main():
    q = asyncio.Queue(maxsize=3)           # bounded → backpressure
    await asyncio.gather(
        producer(q, 6),
        consumer(q, "C1"),
        consumer(q, "C2"),
    )

asyncio.run(main())
Sample output (two consumers share the work)
produced 0
produced 1
produced 2
  C1 handled 0
  C2 handled 1
produced 3
produced 4
  C1 handled 2
  ...
Try this: drop maxsize to unbounded (remove it) and make the consumers much slower than the producer. With a bound, the producer naturally waits; unbounded, it races ahead and the queue grows without limit. This is exactly how you protect a streaming server's memory.

8. TaskGroups & Structured Concurrency (3.11+)

"Structured concurrency" is a fancy name for a homely, sensible rule: work you start inside a block must finish before you leave the block. No tasks sneaking off to run (and fail silently) after the function that spawned them has returned. It's the same discipline with brought to files — open inside, guaranteed closed on exit.

The deal a TaskGroup makes you: "Start as many child tasks as you like inside my async with block. When the block ends, I guarantee they're all done. And if any one of them fails, I'll cancel the others and report every error — no leaks, no surprises."
async def handle_request(req):
    async with asyncio.TaskGroup() as tg:
        t1 = tg.create_task(fetch_user(req.user_id))
        t2 = tg.create_task(fetch_history(req.user_id))
        t3 = tg.create_task(call_llm(req.prompt))
    # We only get here once ALL three are done.
    return assemble(t1.result(), t2.result(), t3.result())

See the auto-cancel in action

Three tasks start. One fails fast. Watch the TaskGroup cancel the survivors instead of leaving them running.

🧪 Experiment 15 — one fails, siblings get cancelled
import asyncio

async def good(name, secs):
    try:
        await asyncio.sleep(secs)
        print(f"{name}: finished")
    except asyncio.CancelledError:
        print(f"{name}: CANCELLED (a sibling failed)")
        raise

async def bad():
    await asyncio.sleep(0.2)
    raise ValueError("bad task exploded")

async def main():
    try:
        async with asyncio.TaskGroup() as tg:
            tg.create_task(good("A", 1.0))
            tg.create_task(good("B", 1.0))
            tg.create_task(bad())
    except* ValueError as eg:        # note the `except*` for ExceptionGroups
        print("caught group:", [str(e) for e in eg.exceptions])

asyncio.run(main())
Predicted output

B: CANCELLED (a sibling failed)
caught group: ['bad task exploded']
The click: With plain gather, A and B would have kept running in the background after bad() exploded — orphaned work burning resources. The TaskGroup treats the three as a family: if one dies, the others are cancelled, and you get a clean ExceptionGroup instead of a half-finished mess.

9. Cancellation, Timeouts, Shielding

Cancelling an async task is polite, not violent. task.cancel() doesn't kill anything on the spot. It leaves a note that says: "the next time you hit an await, raise CancelledError there." The task gets a chance to clean up — close files, roll back, log — before it actually stops.

Watch a cancellation land at the next await

🧪 Experiment 16 — cancel, clean up, stop
import asyncio

async def worker():
    print("worker: starting")
    try:
        for i in range(10):
            print(f"worker: step {i}")
            await asyncio.sleep(0.2)     # cancellation will strike here
    except asyncio.CancelledError:
        print("worker: got cancel → cleaning up")
        await asyncio.sleep(0.1)         # cleanup can itself await
        print("worker: cleanup done")
        raise                            # ALWAYS re-raise

async def main():
    t = asyncio.create_task(worker())
    await asyncio.sleep(0.5)             # let it run a couple steps
    print("main: cancelling worker")
    t.cancel()
    try:
        await t
    except asyncio.CancelledError:
        print("main: worker is fully cancelled")

asyncio.run(main())
Predicted output
worker: starting
worker: step 0
worker: step 1
worker: step 2
main: cancelling worker
worker: got cancel → cleaning up
worker: cleanup done
main: worker is fully cancelled
The golden rule: catch CancelledError only to clean up, then re-raise it. If you swallow it (except asyncio.CancelledError: pass), the loop thinks the task refused to die, and structured concurrency / TaskGroups break in confusing ways. Use try/finally for cleanup that must always run.

Timeouts the modern way (3.11+)

asyncio.timeout is a context manager that cancels whatever's inside if it overruns — cleaner than wrapping every call in wait_for.

async with asyncio.timeout(5):
    result = await call_llm(prompt)      # cancelled + TimeoutError if > 5s

Mentally: it starts a 5-second fuse; if the block isn't done when the fuse burns out, it cancels the current task (so cleanup runs) and raises TimeoutError outward.

Shielding: "cancel me, but not this part"

Sometimes a piece of work must finish even if the caller gives up — committing a payment, flushing a write. Wrap it in asyncio.shield so cancellation can't interrupt it.

# Even if our caller cancels us, the DB commit runs to completion.
await asyncio.shield(commit_to_db(record))
Try this: in Experiment 16, remove the raise at the end of the except block. Re-run — main now prints "worker is fully cancelled" is replaced by the task appearing to complete normally. You've just seen why swallowing CancelledError is dangerous: the cancellation silently vanished.

3.11 niceties: you can attach a reason — task.cancel("user pressed stop") — and read it from the error's args. Task.uncancel() lets advanced code (like a retry layer) say "I've handled that cancellation, it's safe to keep going."

10. Locks, Semaphores, Queues, Events

Here's the reassuring part: in async you need far less locking than in threads. Why? Because your code can only be interrupted at an await. Any stretch of code without an await runs start-to-finish with nobody else butting in. So you only need synchronization when shared state is touched across an await.

First, prove a race condition CAN happen across an await

Two tasks do "read counter → await → write counter+1." Because they both read the same old value before either writes, updates get lost.

🧪 Experiment 17 — a lost update across await
import asyncio

counter = 0

async def increment_unsafe():
    global counter
    tmp = counter            # read
    await asyncio.sleep(0)   # ← yield! the other task sneaks in here
    counter = tmp + 1        # write a stale value

async def main():
    global counter
    counter = 0
    await asyncio.gather(*(increment_unsafe() for _ in range(1000)))
    print("unsafe counter:", counter)   # NOT 1000

asyncio.run(main())
Typical output — updates were lost
unsafe counter: 2     (or some small number, not 1000)

Now wrap the read-modify-write in a Lock so only one task is inside the critical section at a time:

🧪 Experiment 18 — a Lock fixes it
import asyncio

counter = 0
lock = asyncio.Lock()

async def increment_safe():
    global counter
    async with lock:             # only one task in here at a time
        tmp = counter
        await asyncio.sleep(0)
        counter = tmp + 1

async def main():
    global counter
    counter = 0
    await asyncio.gather(*(increment_safe() for _ in range(1000)))
    print("safe counter:", counter)   # exactly 1000

asyncio.run(main())
Predicted output
safe counter: 1000
The click: the bug only exists because there's an await between the read and the write. Remove that await and the unsafe version is actually correct — proof that in async you guard across await boundaries, nothing more.

The semaphore — your most-used async tool in practice

A Semaphore(N) is a bouncer that lets at most N tasks into the club at once; the rest wait at the door. This is how you fan out 10,000 jobs but only hammer an API with, say, 5 at a time.

🧪 Experiment 19 — cap concurrency at 3
import asyncio

active = 0
sem = asyncio.Semaphore(3)      # at most 3 inside at once

async def call_api(i):
    global active
    async with sem:
        active += 1
        print(f"task {i:2d} running   (active now = {active})")
        await asyncio.sleep(0.5)
        active -= 1

async def main():
    await asyncio.gather(*(call_api(i) for i in range(10)))

asyncio.run(main())
Predicted output — 'active' never exceeds 3
task  0 running   (active now = 1)
task  1 running   (active now = 2)
task  2 running   (active now = 3)
task  3 running   (active now = 3)
task  4 running   (active now = 3)
...   (10 tasks total, but never more than 3 at the same time)
Try this: change Semaphore(3) to Semaphore(10) — now all ten run at once and finish in one 0.5s wave. Set it to Semaphore(1) and you've turned concurrency back into a sequential queue. The number is your concurrency dial.
PrimitivePlain-English job
asyncio.Lock"Only one task in this critical section at a time" (e.g., refreshing a shared token).
asyncio.Semaphore(N)"At most N at once" — the concurrency throttle for API calls.
asyncio.Event"Everyone wait until I flip this switch" (model loaded, shutdown requested).
asyncio.Condition"Wait until some predicate becomes true" — rare; a Queue is usually nicer.
asyncio.QueueProducer/consumer hand-off with optional backpressure.
Remember: never fan out 10,000 raw API calls with a bare gather. Always put a Semaphore in front, or you'll flood the provider, trip rate limits, and exhaust file descriptors in seconds.

11. Blocking Code, the GIL, and Escape Hatches

This is the section that fixes 90% of "my async app is mysteriously slow" problems. The whole event loop lives in one thread. If any task calls something that blocks that thread — and doesn't await — the entire app freezes until it returns. Async can't save you from code that won't yield.

Feel the freeze: time.sleep vs asyncio.sleep

They look like twins. One is a polite "park me and run others"; the other is "everybody stop, I'm holding the thread hostage."

🧪 Experiment 20 — the impostor that freezes everything
import asyncio, time

async def heartbeat():
    for i in range(10):
        print(f"  ❤ beat {i}")
        await asyncio.sleep(0.1)

async def blocking_call():
    print("blocking: about to time.sleep(0.5) — NOT awaited")
    time.sleep(0.5)            # ☠ blocks the WHOLE loop
    print("blocking: done")

async def main():
    await asyncio.gather(heartbeat(), blocking_call())

asyncio.run(main())
Predicted output — the heartbeat stalls for 0.5s
  ❤ beat 0
blocking: about to time.sleep(0.5) — NOT awaited
blocking: done            ← everything froze here for half a second
  ❤ beat 1
  ❤ beat 2
  ...

The heartbeat wanted to beat every 0.1s. But time.sleep(0.5) held the only thread, so beats 1–4 couldn't fire until it returned. Now swap in the fix:

🧪 Experiment 21 — offload the blocking call to a thread
import asyncio, time

async def heartbeat():
    for i in range(10):
        print(f"  ❤ beat {i}")
        await asyncio.sleep(0.1)

async def offloaded_call():
    print("offloaded: running time.sleep in a worker thread")
    await asyncio.to_thread(time.sleep, 0.5)   # ✅ loop stays free
    print("offloaded: done")

async def main():
    await asyncio.gather(heartbeat(), offloaded_call())

asyncio.run(main())
Predicted output — heartbeat keeps beating

  ❤ beat 1
  ❤ beat 2
  ❤ beat 3
  ❤ beat 4
offloaded: done
  ❤ beat 5
  ...
The click: asyncio.to_thread(fn, *args) shoves a blocking function onto a background OS thread and hands you an awaitable. The event loop is free again, the heartbeat keeps ticking, and you collect the result when it's done. This is your escape hatch for any stubborn blocking library (legacy requests, sync DB drivers, file reads).

Two escape hatches, and when to use which

# For blocking I/O (sync HTTP, file reads, legacy DB drivers):
data = await asyncio.to_thread(requests.get, url)

# For heavy CPU work that must use multiple cores:
from concurrent.futures import ProcessPoolExecutor
loop = asyncio.get_running_loop()
with ProcessPoolExecutor() as pool:
    result = await loop.run_in_executor(pool, cpu_heavy_fn, big_input)

Rule of thumb: threads for blocking I/O, processes for CPU. Why not threads for CPU? Because of the GIL — next.

The GIL, in plain words

The Global Interpreter Lock is a single "talking stick" inside the Python process: only the thread holding it may run Python bytecode at any instant. Consequences:

  • Threads don't speed up pure-Python CPU work — they take turns holding the stick, so two CPU-bound Python threads aren't faster than one. Use processes (each has its own GIL) for that.
  • Threads still help with blocking I/O — because the stick is released during blocking C calls (socket reads, file ops, NumPy crunching). That's why to_thread works for I/O.
  • Async doesn't fight the GIL at all — it's single-threaded by design. Its concurrency comes from overlapping waiting, which doesn't need the stick.
  • Python 3.13+ ships an experimental free-threaded ("no-GIL") build that removes this limitation, but it's still maturing.
The classic question: "If Python has a GIL, why is async useful?" → Because async never tries to run Python on two cores at once. It overlaps idle waiting (network, disk), and the GIL is happily released during all that waiting. Different problem, no conflict.

12. Async Generators & Context Managers

So far our async functions returned one value. But what if a result arrives in pieces over time — like an LLM streaming tokens, or rows trickling from a database cursor? That's what async generators are for: a function that yields values one at a time, pausing at await between them, and you consume it with async for.

Build a token streamer you can actually run

No network needed — we fake an LLM dripping out one word at a time. Notice the consumer prints each token the instant it's produced, not after all of them are ready.

🧪 Experiment 22 — streaming tokens with async for
import asyncio

async def stream_tokens(sentence):
    for word in sentence.split():
        await asyncio.sleep(0.2)     # pretend each token comes from the network
        yield word                   # hand it out the moment it's ready

async def main():
    async for token in stream_tokens("async makes streaming feel natural"):
        print(token, end=" ", flush=True)   # appears one word at a time
    print()

asyncio.run(main())

Async context managers — guaranteed setup & teardown

You already know with open(...) as f: guarantees the file closes. The async version, async with, does the same for things whose open/close themselves need to await — database connections, HTTP clients, tracing spans.

🧪 Experiment 23 — async with, the easy way
import asyncio
from contextlib import asynccontextmanager

@asynccontextmanager
async def db_connection(name):
    print(f"open {name}")          # __aenter__
    await asyncio.sleep(0.1)       # connecting can await
    try:
        yield f"<conn {name}>"      # this is what `as conn` receives
    finally:
        print(f"close {name}")     # __aexit__ — always runs, even on error
        await asyncio.sleep(0.1)

async def main():
    async with db_connection("primary") as conn:
        print("using", conn)

asyncio.run(main())
Predicted output

using <conn primary>
close primary
Try this: raise an exception inside the async with body (raise RuntimeError("boom")). Notice close primary still prints — the finally in the context manager guarantees cleanup even when things blow up. That's the whole reason to use it.

The async protocol cheat-sheet

  • async for drives __aiter__ / __anext__ (async generators implement these for you).
  • async with drives __aenter__ / __aexit__ (the decorator above writes them for you).
  • Async comprehensions exist too: [x async for x in gen() if await keep(x)].

13. Patterns Every AI Engineer Should Own

These are the recipes you'll actually reach for when wiring LLMs, embeddings, and agents. Each is just the primitives from §7–§10 combined.

13.1 Bounded fan-out for batch inference

"Embed 50,000 documents" — but the API allows only N concurrent calls. Fan out everything, throttle with a semaphore (exactly Experiment 19, applied for real).

async def embed_all(texts, concurrency=20):
    sem = asyncio.Semaphore(concurrency)
    async def one(t):
        async with sem:                       # throttle to `concurrency` at a time
            return await client.embeddings.create(input=t, model="text-embedding-3-small")
    return await asyncio.gather(*(one(t) for t in texts))

13.2 Retry with exponential backoff & jitter

APIs return 429 Too Many Requests under load. Don't hammer — back off, doubling the wait each time, plus a little randomness ("jitter") so a thousand clients don't all retry on the same beat.

import random
async def with_retry(coro_fn, attempts=5, base=0.5):
    for i in range(attempts):
        try:
            return await coro_fn()
        except (RateLimitError, httpx.TimeoutException):
            if i == attempts - 1:
                raise
            delay = base * (2 ** i) + random.uniform(0, base)   # 0.5s, 1s, 2s, 4s …
            await asyncio.sleep(delay)

13.3 Race two providers, take the first to answer

Send the same prompt to two providers; return whichever responds first; cancel the loser so you stop paying for it.

async def race(*coros):
    tasks = [asyncio.create_task(c) for c in coros]
    done, pending = await asyncio.wait(tasks, return_when=asyncio.FIRST_COMPLETED)
    for p in pending:
        p.cancel()
    return done.pop().result()

answer = await race(openai_call(p), anthropic_call(p))

See a realistic pipeline run end to end

This self-contained experiment combines fan-out + semaphore throttle + retry-on-flaky-failure — the shape of a real batch-inference job. Some "API calls" randomly fail once and succeed on retry.

🧪 Experiment 24 — throttled, retrying batch job
import asyncio, random, time

sem = asyncio.Semaphore(3)            # max 3 concurrent "API calls"

async def flaky_api(doc):
    # fails ~40% of the time on first try
    if random.random() < 0.4:
        raise RuntimeError("429 rate limited")
    await asyncio.sleep(0.3)
    return f"embedding({doc})"

async def call_with_retry(doc, attempts=4):
    for i in range(attempts):
        try:
            async with sem:                   # throttle
                return await flaky_api(doc)
        except RuntimeError:
            if i == attempts - 1:
                raise
            await asyncio.sleep(0.1 * 2 ** i)  # backoff
            print(f"  retry {doc} (attempt {i+2})")

async def main():
    docs = [f"doc{i}" for i in range(6)]
    start = time.perf_counter()
    results = await asyncio.gather(*(call_with_retry(d) for d in docs))
    print("results:", results)
    print(f"done in {time.perf_counter()-start:.1f}s")

asyncio.run(main())

  retry doc4 (attempt 2)

13.4 Streaming with backpressure

Wrap a token stream in a bounded Queue. If the downstream consumer (e.g. SSE to a slow browser) falls behind, the producer naturally pauses on put instead of buffering unbounded tokens in memory. Essential for high-traffic chat servers.

13.5 Agent loops with concurrent tool calls

The LLM plans several independent tool calls; run them together in a TaskGroup so they overlap, then synthesize.

async def agent_step(state):
    plan = await llm.plan(state)
    async with asyncio.TaskGroup() as tg:                 # all tool calls overlap
        results = {name: tg.create_task(tools[name](args))
                   for name, args in plan.tool_calls}
    return llm.synthesize(state, {n: t.result() for n, t in results.items()})

13.6 Server frameworks

FastAPI/Starlette/aiohttp endpoints are coroutines running on a shared loop. Never call time.sleep, requests, or a sync DB driver inside an async def endpoint — it freezes the whole server (Experiment 20!). Use asyncio.to_thread or the async-native libs (httpx, asyncpg, aiomysql, motor).

13.7 Graceful shutdown

stop = asyncio.Event()
loop.add_signal_handler(signal.SIGTERM, stop.set)

async def main():
    server = await start_server()
    await stop.wait()                    # block until SIGTERM flips the Event
    await server.drain(timeout=30)       # let in-flight requests finish

14. Anti-Patterns & Footguns

Every one of these is a real bug people ship. You've now seen the experiments behind most of them — here they are as a quick "don't do this" checklist.

1. Forgetting await. fetch(url) returns a coroutine that never runs (Experiment 7). Python warns: RuntimeWarning: coroutine 'fetch' was never awaited.
2. Blocking the loop. time.sleep, requests, open().read() on big files, json.loads on huge payloads, regex with catastrophic backtracking, CPU-heavy pydantic validation. Profile with asyncio's slow-callback warning: set loop.slow_callback_duration = 0.1 in dev.
3. Bare create_task without holding a reference. The loop only holds weak refs to tasks. A "fire and forget" task can be garbage-collected mid-flight. Either await it, store it in a set, or use a TaskGroup.
4. Swallowing CancelledError. Always re-raise. Use try/finally for cleanup, not except: pass.
5. asyncio.run inside an already-running loop. Raises RuntimeError. Common when calling library code from Jupyter (which already has a loop). Use await directly or nest_asyncio as a last resort.
6. Sharing one HTTP client across event loops. Connections are bound to their loop. Create one client per loop; in FastAPI, manage it via lifespan/startup events.
7. Unbounded gather. 100k coroutines = 100k open sockets = file-descriptor exhaustion + provider rate-limit explosion. Use a semaphore.
8. CPU work in a coroutine "because async is fast." Async won't make matrix multiplication faster; it will only hide the latency of waiting. Use a process pool.
9. async generator + asyncio.gather confusion. You can't gather a generator; you must materialize tasks first.
10. Mutating shared state across awaits without a lock. Code is atomic between awaits, not across them. Two coroutines can interleave at any await point and corrupt shared mutable state.

15. Debugging & Observability

  • Debug mode: asyncio.run(main(), debug=True) or env PYTHONASYNCIODEBUG=1. Warns about slow callbacks, never-awaited coroutines, sync calls.
  • Slow callback detection: loop.slow_callback_duration = 0.05 logs any callback exceeding 50ms.
  • Inspect tasks: asyncio.all_tasks() and task.get_stack() to see where every task is suspended.
  • Tracebacks: ExceptionGroups (3.11) make multi-failure stacks readable.
  • Tracing: OpenTelemetry's opentelemetry-instrumentation-asyncio spans across await points; contextvars propagate context automatically.
  • Profiling: py-spy works on async (it samples the call stack). For per-coroutine timing, instrument with decorators that record time.perf_counter() deltas across awaits.

Context variables — the underused trick

from contextvars import ContextVar
request_id: ContextVar[str] = ContextVar("request_id")

async def handler(req):
    request_id.set(req.id)
    await downstream()   # `request_id` is propagated through every await

async def downstream():
    log.info("processing", extra={"req": request_id.get()})

Unlike thread-locals, contextvars are copied per-Task, so concurrent requests never bleed context.

16. Q&A Appendix

What's the difference between concurrency and parallelism, and where does async fit?
Concurrency is a structuring technique — composing a program as independent tasks that may interleave. Parallelism is simultaneous execution on multiple cores. Async is a style of concurrency: cooperative, single-threaded, event-loop driven, ideal for I/O-bound work where most time is spent waiting. Async gives you concurrency without parallelism — exactly what you want for fanning out 1,000 LLM API calls.
Generators vs. coroutines vs. async generators — when do I loop, and when do I await?
All three are pausable functions that build an object and run nothing until driven. Generator (def + yield): calling it returns a generator object; drive it with for x in gen() or next(gen). Coroutine (async def + await): calling it returns a coroutine object; you do not loop over it — you await coro() (or schedule it with create_task). Writing for i in coro() is a category error: a plain coroutine isn't iterable. Async generator (async def + yield): the hybrid — it yields multiple values and awaits between them, so you drive it with async for x in agen(). Rule of thumb: one value → await; a stream of values → async for; a sync stream → plain for.
Walk me through what happens, step-by-step, when Python hits an await.
The expression after await is evaluated to an awaitable. The current coroutine's frame is saved. Control returns to the event loop with a registration "wake me when this awaitable completes." The loop runs other ready tasks and calls selector.select() to wait on OS-level events (epoll/kqueue/IOCP). When the awaited operation completes, the loop puts our coroutine back on the ready queue and resumes it right after the await, with the result substituted in. Mechanically this rides on the generator protocol — coroutines are essentially typed generators driven by the loop.
If Python has a GIL, why is asyncio useful at all?
The GIL serializes Python bytecode but is released during blocking C calls — including socket I/O. Async doesn't try to run Python on multiple cores; it overlaps waiting. While one coroutine is blocked on a socket read inside the kernel, the loop runs others. So the GIL is irrelevant: async's win comes from doing one syscall (epoll_wait) to monitor thousands of sockets, with effectively zero per-task overhead. For CPU-bound work async doesn't help — you need processes or native code.
What's the difference between await coro, create_task(coro), and gather(*coros)?
await coro runs the coroutine inline — sequential. create_task schedules it on the loop so it runs concurrently with whoever called it; you can await the returned Task later to join. gather is a convenience that wraps multiple awaitables, schedules them concurrently, and returns when all are done (or one fails). In modern code prefer TaskGroup over gather for structured cancellation and ExceptionGroup error handling.
Explain structured concurrency and why TaskGroup is preferred over gather.
Structured concurrency: any concurrent work spawned inside a scope must finish before that scope exits. TaskGroup enforces this with a context manager — if any child task raises, all siblings are cancelled and errors are collected into an ExceptionGroup. With plain gather, a first-exception failure leaves the other tasks running in the background (leaks), and cancellation of the parent doesn't reliably propagate. TaskGroup gives lexical scoping for concurrency, the same way with gave it to resources.
How does cancellation work, and what are the pitfalls?
Cancellation is cooperative: task.cancel() schedules a CancelledError to be raised at the task's next await. The task can catch it for cleanup but should re-raise. Pitfalls: swallowing CancelledError breaks structured concurrency; long CPU loops with no await are uncancellable; shielded sections with asyncio.shield intentionally survive cancellation (use for atomic commits). Python 3.11 added asyncio.timeout and Task.uncancel for clean composability.
You added async everywhere but the app got slower. What would you check?
Top suspects, in order: (1) Blocking calls inside coroutines — requests, time.sleep, sync DB drivers, heavy JSON parsing — freeze the loop. Enable debug=True and slow_callback_duration. (2) Sequential awaits in a loop instead of gather/TaskGroup. (3) Unbounded concurrency causing rate-limiting or socket exhaustion — add a semaphore. (4) Creating a new HTTP client per call, losing connection pooling. (5) CPU-bound work that needs a process pool. (6) Calling asyncio.run in a hot path, creating and tearing down a loop each invocation.
How do you bound concurrency when calling an LLM API for 50,000 documents?
A bounded semaphore around each call, plus retry with jittered exponential backoff for 429s, plus an outer TaskGroup for structured cancellation. Optionally a token-bucket rate limiter to respect TPM/RPM quotas. Stream results out via an asyncio.Queue so memory doesn't balloon — backpressure naturally pauses producers when consumers (or a writer to disk/Parquet) fall behind. Persist progress checkpoints so a crash doesn't lose work.
What is the function color problem?
"Async colors a function" — to await something you must be inside an async def, which means every caller up the stack must also be async. Sync and async APIs don't compose without bridging (asyncio.run, to_thread, asgiref.sync_to_async). This is a real cost: libraries duplicate APIs (e.g., requests vs httpx; psycopg2 vs asyncpg) and migrations are viral.
How do contextvars work, and why are they better than thread-locals in async code?
contextvars.ContextVar stores values per Context, and each Task runs in its own copy of the parent's Context. So values like request IDs, tenant IDs, or tracing spans propagate correctly through every await and into spawned tasks without leaking between concurrent requests. Thread-locals fail in async because many tasks share the same OS thread — they'd see each other's data.
How does asyncio actually wait on thousands of sockets in a single thread?
It uses a kernel-level event-notification mechanism: epoll on Linux, kqueue on BSD/macOS, IOCP on Windows. A single epoll_wait syscall blocks until any of the registered file descriptors becomes readable/writable, then returns the set of ready FDs. Asyncio's selectors module abstracts this. So "10,000 concurrent connections" is really "one syscall watching 10,000 FDs."
How do you race two LLM providers and take the first response?
Create a task per provider; use asyncio.wait(..., return_when=FIRST_COMPLETED); cancel the loser. Wrap in asyncio.shield the parts you don't want cancelled mid-write. Be careful with billing — both providers will be charged for at least the request initiation. For LLMs you can also race streaming: read the first token from each, commit to whichever streams first, cancel the other.
You see RuntimeWarning: coroutine 'foo' was never awaited. What does that mean and how do you fix it?
You called an async def function but never awaited or scheduled the returned coroutine, so it never executed and was garbage-collected. Fix by either await foo() if you need its result, or asyncio.create_task(foo()) if you want it to run concurrently — and hold a reference (or use a TaskGroup) so it isn't GC'd.
What's asyncio.to_thread for and when shouldn't you use it?
It runs a sync function in the default thread pool and returns an awaitable, so you can call blocking I/O from async code without freezing the loop. Don't use it for CPU-bound Python — the GIL serializes execution and you'll just clog the thread pool. For CPU work, use run_in_executor with a ProcessPoolExecutor. Also don't over-use it as an escape hatch; native async libraries (httpx, asyncpg) are dramatically more efficient.
Streaming an LLM response to a browser via SSE — describe the async design.
An async generator wraps the provider's stream (httpx.AsyncClient.stream or the SDK's async iterator). The web framework (FastAPI's StreamingResponse) drives async for over it, writing each chunk to the client. Add: a bounded queue if you want to decouple producer and consumer rates, an asyncio.timeout per chunk to detect stalls, a cancellation handler (client disconnect → cancel upstream call to stop billing), and a final finally block that closes the upstream connection. Wrap the whole handler in a try/except for CancelledError to log clean disconnects.
Why might await asyncio.sleep(0) appear in production code?
It's a cooperative yield — it suspends the current task and immediately re-schedules it, giving every other ready task a chance to run. Useful inside a tight loop that has no other awaits (rare in production) or to break up a burst of synchronous-ish work into chunks. In well-designed async code it's a code smell; you should usually be awaiting real I/O.
What's the difference between asyncio, trio, and anyio?
asyncio is stdlib; battle-tested, large ecosystem, historically loose around cancellation. trio pioneered structured concurrency (nurseries) with a strict, principled design — cancellation just works. anyio is a compatibility layer that gives you trio-style APIs running on either backend, which is why FastAPI/Starlette use it. For new code, asyncio + TaskGroup (3.11+) covers most needs; anyio is great for library authors who want backend-agnostic code.
How would you implement a token-bucket rate limiter in asyncio?
An async class holding tokens: float and last_refill: float. An acquire(n=1) coroutine computes elapsed time, refills tokens at rate per second up to capacity, and if there aren't enough tokens, await asyncio.sleep(needed/rate) and retries. Protect the state with an asyncio.Lock if multiple tasks call concurrently. Combine with a semaphore for hard concurrency caps. Production: persist counters to Redis for multi-process.
You have an async function that occasionally hangs forever. How do you debug it?
Enable debug mode (PYTHONASYNCIODEBUG=1). Dump live tasks: for t in asyncio.all_tasks(): t.print_stack() — this shows exactly which await each task is parked at. Wrap suspect calls in asyncio.timeout to convert hangs into errors. Use py-spy dump on the live process for a thread-level view. Look for missing timeouts on network calls, deadlocks on locks held across awaits, and tasks awaiting events that are never set. Add structured logging with contextvars to correlate.
How does FastAPI decide whether to run an endpoint in the event loop or a thread?
If the endpoint is async def, it's awaited directly on the event loop. If it's a plain def, FastAPI runs it in a thread pool (via anyio) so a blocking call inside doesn't freeze the loop. Implication: never put blocking I/O inside an async def endpoint without to_thread — you'll silently kneecap the entire process. If the endpoint mostly does blocking work, leave it sync and let the framework handle the offload.
What's the deal with ExceptionGroup and except*?
PEP 654 (3.11). A TaskGroup can fail with multiple errors simultaneously — collecting them into a single exception was awkward. ExceptionGroup wraps them; except* lets you match by type and handle a subset, with unmatched exceptions re-raised as a smaller group. It makes structured concurrency's "any sibling can fail" model ergonomic.

Three rules to live by:
  1. Never block the loop. If you must, wrap it in to_thread.
  2. Never fan out without bounding (Semaphore) and timing out (asyncio.timeout).
  3. Never spawn a task you don't structurally own (TaskGroup).

The Mindset

By-rote async users see await as "the thing that makes my code work." The deeper read is "the exact moment my task hands control back to the event loop, with everything that implies for cancellation, error propagation, fairness, and shared state." Once that single sentence is muscle memory, every pattern in this guide — TaskGroups, semaphores, shielding, structured concurrency — is just a consequence.
